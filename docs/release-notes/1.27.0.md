## Istio Updated to Version 1.29

We've updated the Istio version to 1.29. See [Announcing Istio 1.29](https://istio.io/latest/news/releases/1.29.x/announcing-1.29/) and the [Istio 1.29 Upgrade Notes](https://istio.io/latest/news/releases/1.29.x/announcing-1.29/upgrade-notes/).

Istio 1.29 introduces three upstream breaking changes. To avoid disruption, the following temporary workarounds are applied to the Kyma Istio installation. These workarounds are intended to be removed in **late July** once consumers have had time to adapt.

### Circuit Breaker Metrics Tracking Disabled by Default

Istio 1.29 disables remaining circuit-breaker capacity metrics by default (`DISABLE_TRACK_REMAINING_CB_METRICS=true`). Prometheus dashboards or alerts that rely on these metrics will see no data after the upgrade.

**Mitigation:** We set `DISABLE_TRACK_REMAINING_CB_METRICS=false` in the istiod deployment to preserve the previous behavior. This workaround will be removed in late July; at that point, remaining circuit-breaker metrics will no longer be emitted by default.

### Debug Endpoint Authorization Enabled by Default

Istio 1.29 enables debug endpoint authorization by default (`ENABLE_DEBUG_ENDPOINT_AUTH=true`). Tools accessing Istio debug endpoints from non-system namespaces (such as Kiali or custom monitoring tools) may receive `403` responses after the upgrade.

**Mitigation:** We set `ENABLE_DEBUG_ENDPOINT_AUTH=false` in the istiod deployment to preserve unrestricted access. This workaround will be removed in late July; at that point, non-system namespaces will be restricted to the `config_dump`, `ndsz`, and `edsz` debug endpoints for same-namespace proxies only.

### HTTP Compression of Envoy Metrics Enabled by Default

Istio 1.29 enables HTTP compression (brotli, gzip, zstd) for the Envoy `prometheus_stats` endpoint by default. Metric scrapers that send an `Accept-Encoding` header but cannot decompress responses may fail to collect metrics. Additionally, the `sidecar.istio.io/statsCompression` pod annotation has been **removed**; workloads using it must migrate to the `proxy.istio.io/config` annotation with `statsCompression: false`.

**Mitigation:** We set `statsCompression: false` globally in `MeshConfig.defaultConfig` to disable compression mesh-wide, matching the previous behavior. This workaround will be removed in late July; at that point, compression will be active by default and `sidecar.istio.io/statsCompression` will no longer be recognized.

##  New Features
- We've added new Cluster Roles for the Istio module and its resources that support Kubernetes native role aggregation. See [#1919](https://github.com/kyma-project/istio/pull/1919).

## Fixed Bugs
- We've fixed an issue where sub-resources were not properly deleted when the Istio CR was deleted. See [#1919](https://github.com/kyma-project/istio/pull/1919).
- We've resolved an issue where the Istio controller used excessive memory during the workload restart phase. See [#1902](https://github.com/kyma-project/istio/issues/1902).
- We've resolved an issue where the Istio controller restarted all proxies continuously when compatibility mode was enabled. See [#1939](https://github.com/kyma-project/istio/issues/1939).

## Enhancements

- We've adapted memory limits for the Istio controller to handle increased load better and prevent out-of-memory terminations. The new limits are set to 4 Gi of memory and 4 CPU cores.
- We've introduced a VerticalPodAutoscaler (VPA) to automatically adjust resource requests and limits based on observed usage.
- We've adapted default tolerations of Istio CNI DaemonSet not to tolerate the `NoSchedule` taint, which is applied to nodes that are not ready.
This change prevents the CNI DaemonSet from being scheduled on such nodes, improving the stability during node rolling updates.
